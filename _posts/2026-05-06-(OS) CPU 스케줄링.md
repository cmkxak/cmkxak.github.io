> 강민철 강사의 「혼자 공부하는 컴퓨터 구조 + 운영체제」 / 유튜브 *운영체제 몰아보기* **11장 CPU 스케줄링**을 직접 보면서 정리한 내용입니다.

## 들어가며

10장에서 *프로세스가 준비 → 실행 → 대기* 상태를 오간다는 걸 봤다.
그렇다면 **준비 상태에서 줄 서 있는 여러 프로세스 중 누구를 먼저 CPU에 올려야 할까?**
이 질문에 답하는 것이 **CPU 스케줄링(CPU Scheduling)** 이다.

스케줄링이 잘못되면, 어떤 프로세스는 응답이 너무 느려지고(반응성↓), 어떤 프로세스는 영원히 차례가 오지 않는다(기아). 반대로 스케줄링이 잘 되면 같은 자원으로도 시스템 처리량(throughput)을 끌어올릴 수 있다.

## 11-1. CPU 스케줄링 개요

### 프로세스 우선순위

모든 프로세스가 똑같이 중요한 건 아니다. OS는 각 프로세스에 **우선순위(priority)** 를 매겨 PCB에 기록해 둔다.

대표적으로 두 부류로 나눌 수 있다.

| 종류 | 특징 | 예시 |
|------|------|------|
| **I/O 집중 프로세스 (I/O-bound)** | 대부분의 시간을 I/O 대기에 쓰고, CPU는 잠깐만 사용 | 디스크 백업, 워드프로세서, 네트워크 서버 |
| **CPU 집중 프로세스 (CPU-bound)** | I/O는 거의 없이 CPU 연산에 집중 | 영상 인코딩, 과학 계산, 컴파일 |

> I/O Bound Process의 우선순위는 CPU-Bound Process의 우선순위 보다 높다.
- 이를 이해하려면 일반적인 프로세스가 어떤 과정을 거치며 실행되는지를 생각해야 한다.
- 대부분의 프로세스는 CPU, 입출력 장치를 모두 사용 하며 실행하는데 이는 실행 상태와 대기 상태를 반복하며 실행한다는 것을 의미한다.
- 입출력 장치는 CPU를 잠깐 쓰고 곧장 I/O 대기로 빠지기 때문에, 먼저 짧게 처리해주면 그 시간 동안 CPU를 다른 프로세스가 알차게 쓸 수 있다.
- 이렇듯, 모든 프로세스가 CPU를 차례로 돌아가며 사용하는 것보다 상황에 맞게 CPU를 배분하는 것이 더 효율적이다.
- 상황에 맞게 CPU를 프로세스들에 배분하기 위해 운영체제는 각 프로세스의 PCB에 우선순위를 부여한다.

### 스케줄링 큐
- PCB에 프로세스의 우선순위가 있지만, CPU를 사용할 프로세스를 찾기 위해 운영체제가 일일이 모든 프로세스의 PCB를 확인하는 것은 비효율적이다.
- 따라서, **스케줄링 큐**로 자원을 사용하고 싶은 프로세스들을 관리한다.

- **준비 큐 (Ready Queue)** — CPU(실행 차례)를 기다리는 프로세스들의 줄
- **대기 큐 (Wait Queue, I/O Queue)** — 입출력 장치 (I/O 작업)를 사용하기 위해 대기 상태에 접어든 프로세스들의 줄 (보통 I/O 장치별로 따로)

```
       ┌──────────────┐
       │  Ready Queue │ ──▶ Dispatch ──▶ CPU 실행 ────┐
       └──────────────┘                              │
              ▲                                      │
              │ I/O 완료                              │ I/O 요청
              │                                      ▼
       ┌──────────────┐
       │   Wait Queue │ ◀────────────────────────────┘
       └──────────────┘
```

- 준비 큐에서 같은 우선순위끼리는 일반적으로 **먼저 들어온 순서**대로 빠진다 (큐의 기본 동작).
- I/O가 끝난 프로세스는 대기 큐에서 빠져나와 다시 **준비 큐의 끝**으로 들어간다.

### 선점형 vs 비선점형 스케줄링

CPU에서 실행 중인 프로세스를 **강제로 끌어내릴 수 있는가?** 가 스케줄링을 가르는 큰 기준이다.

#### 선점형 스케줄링 (Preemptive)
- 실행 중인 프로세스로부터 CPU를 **빼앗을 수 있다**.
- 타이머 인터럽트 혹은 더 높은 우선순위 프로세스의 도착으로 즉시 교체된다.
- 장점: 어느 한 프로세스가 CPU를 독점하지 못함 → **응답성 좋음**.
- 단점: 문맥 교환이 잦아 **오버헤드↑**.
- 예: **Round Robin, SRT, 선점형 우선순위 스케줄링**.

#### 비선점형 스케줄링 (Non-preemptive)
- 실행 중인 프로세스가 **자발적으로 CPU를 놓을 때**(종료 / I/O 대기)까지 기다린다.
- 장점: 문맥 교환이 적어 **오버헤드↓**.
- 단점: 한 프로세스가 길게 잡으면 다른 모든 프로세스가 **기다려야 함**.
- 예: **FCFS, SJF, 비선점형 우선순위 스케줄링**.

> 현대 OS는 대부분 선점형이다.

## 11-2. CPU 스케줄링 알고리즘

### ① FCFS (First Come First Served, 선입선처리)
- CPU를 먼저 요청한 프로세스부터 CPU를 할당하는 스케줄링 방식
- 가장 공정해 보이지만, 프로세스의 사용 시간에 따라 **호위 효과**을 유발함

```
도착 순서:  P1(20초)  P2(2초)  P3(2초)
실행:       [─────── P1 ───────][P2][P3]
                    20s       22s   24s
```

P1이 길어서 P2·P3가 한참을 기다린다. 이런 현상을 **호위 효과(Convoy Effect)** 라 한다.
실행 시간이 짧은 작업이 긴 작업 뒤에 줄 서는 순간, **평균 대기시간이 폭증한다**.

### ② SJF (Shortest Job First, 최단 작업 우선)

- 기본은 **비선점형** (선점형 변형이 SRT).
- **CPU 사용 시간이 가장 짧은 프로세스**부터 실행.
- 평균 대기시간을 이론상 가장 짧게 만들 수 있다. (호위 효과 방지)

```
P1(20초), P2(2초), P3(2초) → 실행 순서: P2 → P3 → P1
```

문제: 긴 작업은 계속 짧은 작업에 밀려 **기아(Starvation)** 가 발생할 수 있다.
또, 사실 OS는 미래에 얼마나 CPU를 쓸지 정확히 모른다 → **예측이 어렵다**.

### ③ 우선순위 스케줄링 (Priority Scheduling)

- 우선순위 값을 기준으로 **높은 쪽부터** 실행.
- SJF도 사실은 "짧은 작업 = 높은 우선순위"인 우선순위 스케줄링의 한 형태다.
- SRT도 사실은 "남은 시간이 짧은 프로세스 = 높은 우선순위" 인 우선순위 스케줄링의 한 형태다.
- **선점형 / 비선점형** 둘 다 가능.

문제: 낮은 우선순위 프로세스의 **기아(Starvation)** 발생.

> 해결책: **에이징(Aging)** — 오래 기다린 프로세스의 우선순위를 시간이 지날수록 점점 올려 준다. 결국 누구든 차례가 오게 보장.

### ④ RR (Round Robin, 라운드 로빈)

- **선점형**.
- 선입 선처리 스케줄링에 타임 슬라이스라는 개념이 더해진 스케줄링 방식
- 타임 슬라이스 : 각 프로세스가 cpu를 사용할 수 있는 시간
- 각 프로세스에게 **타임 슬라이스(time slice / time quantum)** 만큼만 CPU를 주고, 다 쓰면 강제로 준비 큐 뒤로 보낸다.

```
타임 슬라이스 = 2초
P1(5)  P2(3)  P3(2)

[P1:2][P2:2][P3:2][P1:2][P2:1][P1:1]
   2    4    6    8    9    10
```

- 타임 슬라이스가 너무 **크면** → FCFS처럼 동작 (빼앗을 일이 거의 없음) -> 호위 효과를 발생시킬 여지가 있음.
- 타임 슬라이스가 너무 **작으면** → 문맥 교환만 잔뜩 일어나 효율이 떨어짐.
- 적당한 값을 잡는 게 핵심.

### ⑤ SRT (Shortest Remaining Time, 최소 잔여 시간)

- **선점형 SJF**.
- 새로운 프로세스가 도착했을 때, **남은 CPU 시간**이 현재 실행 중인 것보다 짧으면 즉시 교체.
- 응답성과 평균 대기시간 모두 좋지만, 역시 긴 작업 **기아 위험**.

### ⑥ 다단계 큐 (Multilevel Queue)

준비 큐 자체를 **여러 개로 분리**하고, 각 큐에 우선순위를 부여한다.

```
높은 우선순위
  ┌────────────────────────┐
  │  Queue 1: 시스템 프로세스 │
  ├────────────────────────┤
  │  Queue 2: 대화형 프로세스 │
  ├────────────────────────┤
  │  Queue 3: 배치 프로세스   │
  └────────────────────────┘
낮은 우선순위
```

- 위 큐가 비어야 아래 큐의 프로세스가 실행될 수 있다.
- 큐 안에서는 각자 다른 알고리즘(예: Q1=RR, Q3=FCFS)을 쓸 수도 있다.
- 단점: **한 번 들어간 큐에서 못 빠져나와** 기아 발생 가능.

### ⑦ 다단계 피드백 큐 (Multilevel Feedback Queue)

> 다단계 큐의 단점을 보완한 **현대 OS에서 가장 널리 쓰이는 방식**.

- 각 큐에 RR을 적용하고, **타임 슬라이스를 다 못 끝낸 프로세스(=CPU bound)** 는 **한 단계 낮은 큐**로 내려보낸다.
- 짧게 끝나는 I/O bound 프로세스는 위 큐에 계속 머물러 **빠르게 응답**받는다.
- 오래 기다린 프로세스에는 **에이징**으로 우선순위를 다시 올려 줘서 기아도 방지.

```
도착 ──▶ Q1 (RR, 슬라이스 4ms)  ─타임아웃─▶ Q2 (RR, 슬라이스 8ms)
                                  ─타임아웃─▶ Q3 (RR, 슬라이스 16ms)
                                  ─타임아웃─▶ Q4 (FCFS, 가장 낮은 우선순위)

(에이징: Qn 에서 너무 오래 기다린 프로세스는 위 큐로 다시 승급)
```

## 알고리즘 한눈에 비교

| 알고리즘 | 선점성 | 핵심 아이디어 | 장점 | 단점 |
|---------|--------|------------|------|------|
| **FCFS** | 비선점 | 먼저 온 순서대로 | 단순 | 호위 효과 |
| **SJF** | 비선점 | 짧은 작업 먼저 | 평균 대기시간 최소 | 기아, 예측 어려움 |
| **SRT** | 선점 | 남은 시간 짧은 작업 우선 | 반응성↑ | 기아 |
| **우선순위** | 둘 다 | 우선순위 높은 순 | 유연 | 기아 → 에이징 |
| **RR** | 선점 | 타임 슬라이스 순환 | 공정, 반응성↑ | 슬라이스 튜닝 필요 |
| **다단계 큐** | 선점 | 큐를 우선순위별로 분리 | 종류별 정책 적용 | 큐 간 이동 X → 기아 |
| **다단계 피드백 큐** | 선점 | 큐 간 이동 + 에이징 | 현실적 균형 | 가장 복잡 |

## 정리

- 준비 큐의 여러 프로세스 중 **누구를 다음 Running으로 보낼지**가 CPU 스케줄링이다.
- 프로세스에는 **우선순위**가 있고, 일반적으로 **I/O 집중 프로세스에 더 높은 우선순위**를 준다.
- OS는 **준비 큐**와 **대기 큐(I/O 별)** 로 프로세스를 관리한다.
- 실행 중인 프로세스의 CPU를 빼앗을 수 있으면 **선점형**, 아니면 **비선점형**.
- 알고리즘은 크게 **FCFS / SJF / SRT / 우선순위 / RR / 다단계 큐 / 다단계 피드백 큐**.
- 단순한 알고리즘은 **호위 효과 · 기아** 같은 문제가 있고, 이를 보완하기 위해 **에이징**과 **다단계 피드백 큐** 가 등장했다.

> 다음 장은 **프로세스 동기화**. 멀티스레드에서 같은 자원에 동시에 접근할 때 생기는 문제와, 그걸 막는 임계 구역·뮤텍스·세마포어를 다룬다.
