## 들어가며

운영체제 입장에서 **프로세스(Process)** 는 *실행 중인 프로그램* 이다.
디스크에 있는 프로그램을 실행하는 순간 메모리에 올라오고, OS가 그 실행을 관리하기 위해 다양한 정보를 갖게 된다.
이 장에서는 OS가 프로세스를 어떻게 구분·관리하고, 그 안에서 동시에 실행되는 단위인 **스레드(Thread)** 가 무엇인지를 다룬다.

## 10-1. 프로세스 개요

### 프로세스 직접 확인하기

모든 운영체제는 현재 실행 중인 프로세스를 확인하는 도구를 제공한다.

```bash
# Linux / macOS
ps -ef

# Windows
tasklist
```

작업 관리자(Windows) / 활성 상태 보기(macOS) / `top`(Linux)에서도 동일하게 볼 수 있다.

#### 포어그라운드 / 백그라운드 프로세스
- **포어그라운드 프로세스 (Foreground Process)**: 사용자가 직접 보고 상호작용하는 프로세스 (예: 메모장, 브라우저)
- **백그라운드 프로세스 (Background Process)**: 사용자가 보지 않는 곳에서 실행되는 프로세스
  - 사용자와 상호작용하는 백그라운드 프로세스
  - 사용자와 상호작용 없이 정해진 일만 수행하는 프로세스 → **데몬(daemon)** / **서비스(service)**

### 프로세스 제어 블록 (PCB)

> CPU는 한 번에 하나의 프로세스만 실행할 수 있는데, 메모리에는 수많은 프로세스가 올라와 있다. OS는 이들을 어떻게 빠르게 번갈아가며 실행할까?

답은 **PCB (Process Control Block)** 다. OS는 각 프로세스마다 PCB라는 자료구조를 만들어 커널 영역에 보관한다.

PCB에 담기는 대표적인 정보:

| 항목 | 설명 |
|------|------|
| **PID** | 프로세스 식별 번호 (Process ID) |
| **레지스터 값** | PC(프로그램 카운터) 등 다음 실행 지점 정보 |
| **프로세스 상태** | 준비/실행/대기 등 (10-2 참고) |
| **CPU 스케줄링 정보** | 언제, 어떤 순서로 CPU를 할당받을지 |
| **메모리 관리 정보** | 베이스 레지스터, 한계 레지스터, 페이지 테이블 등 |
| **사용한 파일·I/O 장치 목록** | 어떤 자원을 점유 중인지 |

PCB는 **프로세스가 생성될 때 만들어지고, 종료되면 폐기**된다.

### 문맥 교환 (Context Switching)

> CPU에는 한 프로세스만 올라가지만, 그 프로세스는 영원히 CPU를 차지하지 않는다.

타이머 인터럽트 또는 입출력 요청 등으로 프로세스가 잠시 CPU에서 내려와야 하는 시점이 오면, OS는 다음 동작을 한다.

1. 현재 프로세스의 상태(레지스터, PC 등)를 PCB에 **백업**한다 → 이때 백업하는 정보를 **문맥(context)** 이라 부른다.
2. 다음으로 실행할 프로세스의 PCB에서 문맥을 **복원**해 CPU에 올린다.

이렇게 하나의 프로세스에서 다른 프로세스로 실행 흐름이 옮겨가는 과정을 **문맥 교환(Context Switching)** 이라고 한다.

> 문맥 교환은 빠르지만 *공짜는 아니다*. 너무 자주 일어나면 정작 일은 못 하고 교환만 하다가 시간을 낭비한다 (오버헤드).

### 프로세스의 사용자 영역 (메모리 구조)

PCB는 *커널 영역* 에 있다. 그러면 프로세스 자신은 *사용자 영역* 의 메모리에 어떻게 자리 잡을까?

```
┌─────────────────────┐  ← 높은 주소
│      스택(Stack)     │   ↓ 위에서 아래로 자란다
├─────────────────────┤
│         …           │
├─────────────────────┤
│      힙(Heap)        │   ↑ 아래에서 위로 자란다
├─────────────────────┤
│   데이터(Data)       │   전역/정적 변수
├─────────────────────┤
│   코드(Code/Text)    │   기계어 명령어 (읽기 전용)
└─────────────────────┘  ← 낮은 주소
```

- **코드 영역 (Text)**: 실행할 기계어. 변경되지 않으므로 *읽기 전용*.
- **데이터 영역 (Data)**: 전역 변수, 정적(static) 변수처럼 프로그램 내내 유지되는 데이터.
- **힙 영역 (Heap)**: 동적으로 할당받는 메모리 (`malloc`, `new`).
- **스택 영역 (Stack)**: 함수 호출 시 만들어지는 매개변수, 지역 변수, 복귀 주소 등.

코드·데이터는 크기가 정해져 있어 **정적 할당 영역**, 힙·스택은 실행하면서 늘었다 줄었다 하는 **동적 할당 영역**이다.

힙과 스택은 서로 마주 보는 방향으로 자라기 때문에, 너무 많이 쓰면 충돌해서 **스택 오버플로(Stack Overflow)** / **힙 오버플로(Heap Overflow)** 가 발생한다.

## 10-2. 프로세스 상태와 계층 구조

### 프로세스 상태 다섯 가지

PCB의 핵심 정보 중 하나가 **프로세스 상태**다. OS는 프로세스를 다음 상태들로 관리한다.

| 상태 | 설명 |
|------|------|
| **생성 (New)** | 메모리에 적재되고 PCB가 만들어진 직후 |
| **준비 (Ready)** | 곧 CPU를 할당받기 위해 차례를 기다리는 상태 |
| **실행 (Running)** | 실제로 CPU를 할당받아 실행 중 |
| **대기 (Blocked / Waiting)** | I/O 요청 등 시간이 걸리는 작업이 끝나기를 기다리는 상태 |
| **종료 (Terminated)** | 실행이 끝나 PCB와 메모리가 정리되는 단계 |

상태 전이 흐름:

```
 New ──admit──▶ Ready ──dispatch──▶ Running ──exit──▶ Terminated
                  ▲                    │
                  │                    ├─ I/O 요청 ──▶ Blocked
                  │                    │
                  └────────timeout─────┘   (타이머 인터럽트 → 다시 Ready)
                                       
            Blocked ──I/O 완료──▶ Ready
```

핵심 규칙:
- **CPU는 항상 하나의 Running 프로세스만** 실행한다.
- 타이머 인터럽트가 걸리면 Running → Ready 로 강제로 내려간다.
- I/O 같이 시간이 오래 걸리는 일은 Blocked 상태로 보내고, 그 사이 다른 프로세스를 Running 으로 올린다.

### 프로세스 계층 구조

운영체제가 부팅되면 가장 먼저 만들어지는 프로세스가 있고(`init`, `systemd` 등), 이후 모든 프로세스는 **이미 존재하는 프로세스가 다른 프로세스를 만들어내는** 방식으로 생성된다.

이때 만든 쪽을 **부모 프로세스(Parent)**, 만들어진 쪽을 **자식 프로세스(Child)** 라 부른다.

PCB에는 자기 자신의 ID인 **PID** 외에, 부모를 가리키는 **PPID(Parent PID)** 가 함께 기록된다.

```
init(1)
├── sshd(1024)
│   └── bash(2050)
│       └── ./myapp(2080)
└── cron(1100)
```

이 트리 구조에서 *형제 프로세스끼리는 같은 부모* 를 가지고, 부모가 종료되면 자식은 보통 다른 프로세스(`init`)에게 입양되거나 함께 정리된다.

> "그러면 자식 프로세스는 어떻게 만들까?" — 이게 다음 절(10-3)의 주제다.

## 10-3. 프로세스 생성 기법

POSIX 계열 OS에서 부모는 두 시스템 콜의 조합으로 자식 프로세스를 만든다.
**fork(복제)** 와 **exec(변신)** 두 가지를 이해하면 끝이다.

### fork — 복제

> 부모 프로세스의 **복사본**을 자식으로 만든다.

```c
pid_t pid = fork();
if (pid == 0) {
    // 여기는 자식 프로세스
} else if (pid > 0) {
    // 여기는 부모 프로세스 (pid = 자식의 PID)
} else {
    // 생성 실패
}
```

자식은 부모의 자원(코드·데이터·힙·스택, 파일 디스크립터, 환경 변수 등)을 그대로 복사받는다.
- **PID와 PPID** 만 다르다.
- 이름 그대로 "복제"이기 때문에 부모와 자식은 **같은 코드**를 실행한다.
- `fork()` 의 반환값으로 자기가 부모인지 자식인지를 구분한다 (0이면 자식).

### exec — 변신

> 자기 자신의 메모리를 새로운 프로그램으로 **갈아엎는다**.

```c
execlp("/bin/ls", "ls", "-l", NULL);
```

`exec` 계열을 호출하면 호출한 프로세스의 **코드·데이터·힙·스택이 새 프로그램의 것으로 덮어씌워진다**.
- 새 PCB가 만들어지지 않는다 — 같은 프로세스가 "내용물만" 바꿔 입는다.
- 호출이 성공하면 그 이후 줄의 코드는 **절대 실행되지 않는다** (이미 메모리가 새 프로그램으로 바뀌었기 때문).

### 둘을 같이 쓰는 이유 — 쉘이 명령어를 실행하는 방식

> "fork 해서 자식을 만들고, 자식이 exec 해서 다른 프로그램으로 변신한다." — 이게 유닉스 계열의 핵심 패턴이다.

```c
pid_t pid = fork();              // 1. 쉘 자신을 복제
if (pid == 0) {
    execlp("ls", "ls", "-l", NULL); // 2. 자식이 ls 프로그램으로 변신
    perror("exec failed");
} else {
    wait(NULL);                   // 3. 부모(쉘)는 자식이 끝날 때까지 대기
}
```

이 한 줄짜리 패턴이 우리가 매일 쓰는 쉘의 동작 그 자체다.

| 단계 | 일어나는 일 |
|------|------------|
| 1. fork | 쉘 프로세스가 자식(쉘 복제본)을 만든다. PCB와 사용자 메모리가 통째로 복제됨. |
| 2. exec | 자식의 메모리만 `ls` 프로그램으로 갈아끼운다. PID는 그대로. |
| 3. wait | 부모는 자식이 끝날 때까지 Blocked 상태로 기다린다. |
| 4. 종료 | 자식이 종료되면 부모가 깨어나 다음 입력을 받는다. |

> 부모가 자식을 만들기만 하고 안 기다리면 — 백그라운드 실행(`./myapp &`)이 된다.

## 10-4. 스레드

### 스레드란?

> **프로세스 안에서 실행되는 흐름의 단위.**

하나의 프로세스는 기본적으로 한 줄기의 실행 흐름(메인 스레드)을 가진다. 여기에 흐름을 더 만들면 **멀티스레드 프로세스**가 된다.

```
 프로세스 P
 ├── Thread 1 (각자의 PC, 레지스터, 스택)
 ├── Thread 2
 └── Thread 3
        ↑
  코드 / 데이터 / 힙 / 파일은 모두 공유
```

### 프로세스와 스레드의 차이

| 구분 | 프로세스 | 스레드 |
|------|---------|--------|
| 자원 공유 | **공유하지 않음** (각자 독립된 메모리) | **공유함** (같은 프로세스 안의 스레드끼리) |
| 생성 비용 | 큼 (fork → 새 PCB, 새 메모리) | 작음 (PC·레지스터·스택만 새로 할당) |
| 통신 비용 | IPC(파이프, 시그널, 공유 메모리 등) 필요 | 그냥 같은 메모리 읽고 씀 |
| 안정성 | 한 프로세스 죽어도 다른 프로세스에 영향 없음 | 한 스레드 죽으면 같은 프로세스 전체가 위험 |

스레드 자기만의 것:
- **PC (프로그램 카운터)**
- **레지스터 값**
- **스택**

스레드끼리 **공유**하는 것:
- **코드 영역**
- **데이터 영역 (전역/정적 변수)**
- **힙 영역**
- 열어둔 파일, 소켓 등

### 멀티프로세스 vs 멀티스레드

같은 작업을 여러 흐름으로 동시에 처리하고 싶을 때 두 가지 선택지가 있다.

#### 멀티프로세스
- 여러 프로세스를 띄워 일을 나눈다.
- 메모리·자원이 완전히 분리되어 있어 **한쪽이 죽어도 다른 쪽이 살아남는다 (안정성↑)**.
- 대신 자원을 따로따로 잡아먹어 **메모리·생성 비용이 비싸고**, IPC 통신이 번거롭다.

#### 멀티스레드
- 한 프로세스 안에 여러 스레드를 둔다.
- 자원(코드·데이터·힙)을 **공유**하므로 메모리 효율이 좋고 통신이 빠르다.
- 단점: **한 스레드가 비정상 종료되면 같은 프로세스 전체에 영향**을 준다. 또한 같은 자원에 동시에 접근하면서 **동기화 문제**가 발생할 수 있다 (다음 장에서 다룸).

> 안전성이 최우선이면 멀티프로세스, 효율성·반응성이 최우선이면 멀티스레드. 실제 시스템은 둘을 섞어 쓰는 경우가 많다 (예: 크롬은 탭마다 멀티프로세스 + 각 프로세스 안에서 멀티스레드).

## 정리

- **프로세스**는 실행 중인 프로그램이고, OS는 이를 **PCB**로 관리한다.
- CPU는 하나만 실행할 수 있으므로 **문맥 교환**으로 여러 프로세스를 빠르게 번갈아 실행한다.
- 프로세스의 사용자 영역 메모리는 **코드 / 데이터 / 힙 / 스택** 으로 나뉜다.
- 프로세스는 **생성 → 준비 → 실행 → 대기 → 종료** 상태를 오가며, 부모-자식의 **계층 구조**를 이룬다.
- 프로세스 생성은 **fork (복제) + exec (변신)** 의 조합으로 이뤄진다.
- **스레드**는 프로세스 안의 실행 흐름이며, 같은 프로세스 안에서는 **자원을 공유**한다.
- 자원 격리가 중요하면 **멀티프로세스**, 효율성이 중요하면 **멀티스레드** — 단, 멀티스레드는 동기화 문제가 따라붙는다 (다음 장 예고).

> 다음 장은 **CPU 스케줄링**. "여러 프로세스가 준비 상태에서 줄 서 있을 때, 누구를 다음 Running으로 보낼 것인가?"가 주제다.
